项目开发流程总结

Created2023-08-20|Updated2023-08-20|项目项目开发流程
|Post View:

@TOC

前言

刚进公司，需要学开发需要的新框架及现成框架，完成自己的需求。那么在此之前就需要学项目开发流程，比如写开发文档【从设计到实现到测试到文档管理】等，当然啦，肯定大家都有一定差异啦，大家作为参考哦，与诸君共勉。

一、项目整体开发流程:需求(务必反串讲)–》结合业务–》出方案【概设、详设等】–》结合技术出实现–》测试-》交付

1.首先理清需求【务必反串讲，给客户总结一下，您看，您的需求是不是这个意思：…，否则做出来的不是想要的或者差太多，就不太行了】：

  • 其次，需求评审【早上跟其他部门一块开会，晚上日清会，周五周会，不就是干这活的吗】
    • 自己大概评一下
      • 有时候可能需要结合公司的各个需求，评估一下多个需求的优先级，因为你干这个活时有可能中途会插入进来新活哦。【比如结合主线、定制等给任务排优先级，和团队进行沟通】
    • 相关管理人员评一下
  • 还有，可行性分析:【自己大概评一下、相关管理人员评一下】【早上跟其他部门一块开会，晚上日清会，周五周会，不就是干这活的吗】
    • 有哪些需求
    • 怎样完成那些需求，用什么技术，
    • 指定工期内能不能用某个技术完成某个需求
      • 例子，经典案例：项目中要使用地图，百度or高德…，人家两千多人开发出来的地图，你一个人两个星期搞出来???，要不要考虑买个模块呀靓仔！

2.结合业务出方案【有时候需要产出文档】：

  • 1.概设

    • 进度安排：
      • （个人职责）【作为核心开发、作为PL、PO、PM等职责是不一样的哦，具体开发实现、核心功能开发】
        • 接口定义，工程结构设计、代码review、各方沟通，核心功能开发【接口设计的原则：能遵循就遵循，但是也不是不遵循这个就实现不了功能，也不是这样。】
          • 比如 接口名字中带上版本号。要用驼峰都用驼峰要用下划线都用下划线
          • 比如 restful风格的接口【restful风格的接口，重在资源，强调资源变现层状态转移，设计接口时围绕资源来设计，】，支持这个风格的就是HTTP协议
          • 比如，接口安全原则等。手机号、身份证号脱敏
    • 项目和产品:矩阵式（平时是开发：1组、2组，产品：1组、2组），来了项目，从开发中抽一个人从产品中抽一个人，组成一个项目组，然后所有前后端、测试等聚一块，什么时间点完成哪个版本的哪些功能
    • 人员安排
    • 成本估计
    • 质量保证计划
      • 代码层面
      • 业务层面
    • 风险
      • 自己预报、团队成员预报

  • 2.详设【结合概设进行】

    • 详设通过之后才能进行编码实现【详细涉及，对于这个需求你要怎样实现，会有什么漏洞和问题，其他人会提出自己的问题，你拿出自己的解决方案，都通过之后进行实现】

  • 3.方案评审【早上跟其他部门一块开会，晚上日清会，周五周会，不就是干这活的吗】

3.结合技术出实现：

  • 此时可以考虑项目中的主要角色，可以分不同层面来说
    • 比如之前的服务区项目，项目有三个端
      • 服务区监管部门，指派任务【指定将要评价的服务区，哪个省哪个市哪个区，哪个服务区的哪个区域】及对最终结果进行评估
      • 服务区工作人员，接受任务并实施，与监管部门进行信息对接
      • 服务区管理人员
    • 或者说可以参考DDD，对代码结构梳理一下
      • 领域驱动设计（Domain-Driven Design，简称DDD）是一种软件开发方法论，旨在帮助开发人员更好地理解和解决复杂业务领域中的问题。在DDD中，领域模型是对业务领域的抽象和建模，它由实体（Entities）、值对象（Value Objects）、聚合根（Aggregate Roots）、领域服务（Domain Services）等组成。实体是具有唯一标识的对象，值对象是没有唯一标识的对象，聚合根是一组相关对象的根，领域服务是处理领域逻辑的操作。参考周志明老师的凤凰架构的一段话：XXX单体架构后端参考（并未完全遵循）了 DDD 的分层模式和设计原则，整体分为以下四层：
          1. Resource：对应D DD 中的 User Interface 层，负责向用户显示信息或者解释用户发出的命令。请注意，这里指的“用户”不一定是使用用户界面的人，可以是位于另一个进程或计算机的服务。由于本工程采用了MVVM 前后端分离模式，这里所指的用户实际上是前端的服务消费者，所以这里以 RESTful 中的核心概念“ 资源” （Resource）来命名。
          1. Application：对应 DDD 中的 Application 层，负责定义软件本身对外暴露的能力，即软件本身可以完成哪些任务，并负责对内协调领域对象来解决问题。根据 DDD 的原则，应用层要尽量简单，不包含任何业务规则或者知识，而只为下一层中的领域对象协调任务，分配工作，使它们互相协作，这一点在代码上表现为 Application 层中一般不会存在任何的条件判断语句。在许多项目中， Application 层都会被选为包裹事务（代码进入此层事务开始，退出此层事务提交或者回滚）的载体。
        • 3.Domain：对应 DDD 中的 Domain 层，负责实现业务逻辑，即表达业务概念，处理业务状态信息以及业务规则这些行为，此层是整个项目的重点。
        • 4.Infrastructure：对应 DDD 中的 Infrastructure 层，向其他层提供通用的技术能力，譬如持久化能力、远程服务通讯、工具集，等等。
    • 代码质量的话，包括但不限于哦
    • 代码规范。统一帕斯卡。注释【C#///生成的不算，逻辑中//的代码注释小于代码量的1/3】、对于大篇if…else、switch想办法优化
      • 代码质量、考虑代码边界、用户额外操作带来的意外、多领域交接干项目容易出现的意外情况、业务隔离、单一职责、代码的健壮性【有时候有的错误不能抛出，空值异常等不能导致产品直接崩】、松耦合，模块间不能耦合的太死，强耦合、

4.测试：

  • 自测
  • 团队代码check

5.交付：

  • 保证时间，按时交付
  • 保证质量【保证质量的方法，去看看，编程经验与编程陷阱，慢慢提高交付质量】

6.用户反馈问题、测试反馈问题，进行闭环修改
